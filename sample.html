NexusShare: A Secure File Sharing and Collaboration Platform

Introduction and Executive Summary

This document provides a comprehensive overview of the design, architecture, and implementation of NexusShare, a full-stack file sharing platform. Developed using the MERN stack (MongoDB, Express.js, React, Node.js), NexusShare is engineered to provide a secure, scalable, and user-centric solution for modern file management and real-time collaboration. The platform addresses the growing need for a robust system that allows individuals and teams to store, manage, and share digital assets securely and efficiently.

The core objective of this project is to create a feature-rich ecosystem that encompasses secure user authentication, intuitive file and folder management, version control, and real-time sharing capabilities powered by WebSockets. By integrating advanced features such as expiring public links and granular user permissions, NexusShare aims to deliver a superior alternative to generic cloud storage solutions, focusing on security and collaboration. This document will detail the project's vision, technical architecture, feature set, and future roadmap.

Project Vision, Goals, and Problem Statement

Vision

Our vision is to create a seamless and secure digital environment where collaboration is not hindered by technical limitations. We envision NexusShare as the go-to platform for professionals, students, and teams who require a reliable and intuitive way to manage their digital workflow, ensuring that their data is both accessible and protected at all times.

Problem Statement

In today's digital landscape, users face several challenges with existing file management solutions. These include fragmented workflows, inadequate security for sensitive documents, difficulties in tracking file versions, and a lack of real-time collaboration features. Many platforms offer generic storage without the specialized tools needed for controlled sharing and collaborative work. NexusShare aims to solve these problems by providing a unified platform that integrates secure storage with powerful sharing and real-time notification features.

Project Goals

  1. Deliver a Secure Platform: Implement end-to-end security, including robust user authentication, password hashing, and secure file access control.
  2. Provide an Intuitive User Experience: Design a clean, responsive, and easy-to-navigate user interface with features like drag-and-drop uploads and a clear file organization structure.
  3. Enable Real-Time Collaboration: Utilize WebSockets to provide instant notifications for file shares and other critical events, fostering a dynamic collaborative environment.
  4. Ensure Scalability and Reliability: Build the application on a scalable architecture that can handle a growing user base and increasing data loads, with a target uptime of 99.9%.

Technology Stack Selection and Justification

The technology stack for NexusShare was chosen to create a high-performance, scalable, and maintainable application, leveraging the power of a unified JavaScript ecosystem.

  • Front-End (React): Chosen for its component-based architecture, which allows for the creation of a modular and reusable user interface. Its virtual DOM ensures a fast and responsive user experience, critical for interactive features like drag-and-drop. React's vast ecosystem and strong community support also reduce development time and provide access to a wide range of libraries.
  • Back-End (Node.js & Express.js): Node.js's non-blocking, event-driven architecture is ideal for handling concurrent connections and real-time events, such as those managed by Socket.io. Express.js provides a minimalist and flexible framework for building the robust RESTful APIs that power the application, allowing for clean and organized route handling and middleware integration.
  • Database (MongoDB): As a NoSQL database, MongoDB offers a flexible, JSON-like document model that maps directly to the application's data structures. This flexibility is perfect for storing varied metadata associated with files and users, and its native support for horizontal scaling (sharding) makes it well-suited for a growing application with large data volumes.
  • Real-Time Communication (Socket.io): This library enables low-latency, bi-directional communication between clients and the server. It is the core technology behind NexusShare's real-time notification system for file shares and collaborative updates, providing a more dynamic user experience than traditional polling methods.

System Architecture Design

NexusShare is built upon a classic client-server architecture, optimized for performance and scalability. The system is logically divided into three primary layers: the frontend client, the backend server, and the database.

Architectural Diagram

  • Client (React): A Single Page Application (SPA) that provides a dynamic and interactive user interface. It communicates with the backend via RESTful API calls for data operations and maintains a persistent WebSocket connection for real-time events.
  • Backend (Node.js/Express): A server that exposes a set of RESTful APIs to handle business logic, including user authentication, file operations, and database interactions. It also manages WebSocket connections with clients to push real-time notifications.
  • Database (MongoDB): The persistence layer where all application data, including user profiles, file metadata, folder structures, and sharing permissions, is stored.

Database Design and Implementation

The MongoDB schema is designed to be flexible and efficient, modeling the core entities of the application: Users, Files, and Folders. The relationships between these entities are managed through references.

Schema Relationships Visualization

Collection Details

  • User Collection: Stores user profile information, including name, email, and a hashed password. Each user is a distinct entity.
  • File Collection: Contains metadata for every uploaded file, such as the original filename, MIME type, size, and path on the server. It holds a reference to the `owner` (a User) and an array of users it is `sharedWith`.
  • Folder Collection: Defines the directory structure. Each folder has a name, an `owner`, and a reference to a `parent` folder, allowing for nested directories.

File Management System

Upload Process

The platform provides an intuitive file upload experience, supporting both drag-and-drop and traditional file selection. On the backend, multer middleware processes multipart/form-data requests, saving files to a designated /uploads directory with a unique system-generated filename to prevent conflicts. The original filename and other metadata are stored in the MongoDB `File` collection.

Version Control

To prevent accidental data loss, NexusShare includes a basic version control system. When a user uploads a file with the same name as an existing file, the system archives the old version by moving its details into a `versions` array within the file's document. This allows for the potential recovery of previous file states.

Public Link System

Users can generate secure public links for sharing files with external collaborators. Each link is associated with a unique, cryptographically secure token. For enhanced security, users can add an optional password and an expiration date to these links, ensuring that access is both controlled and time-limited.

Frontend Architecture

React Context Providers

Global state is managed efficiently using React's Context API, avoiding the need for prop drilling. Two primary contexts are used:

  • AuthContext: Manages the user's authentication state, including the user object and JWT. It provides `login`, `register`, and `logout` functions to all components.
  • NotificationContext: Manages real-time notifications received via Socket.io, handling the display of toast messages and a list of recent notifications.

Component Structure

The frontend is organized into a logical structure of reusable components, categorized by feature:

  • Dashboard Components: Core UI for file management, including file lists, upload modals, and search bars.
  • Authentication Components: Forms and logic for user login and registration.
  • Sharing Components: Modals for sharing files with other users and generating public links.
  • Layout Components: The main navigation bar and notification display elements.

Real-time Features with Socket.io

The real-time collaborative aspect of NexusShare is powered by Socket.io, which enables persistent, bi-directional communication between the client and server.

Socket.io Integration

Upon logging in, each client establishes a WebSocket connection with the server and joins a personal "room" identified by their unique user ID. This allows the server to target specific users with notifications.

Notification System

When a user shares a file, the backend emits a `file-shared` event to the recipient's personal room. The recipient's client listens for this event and triggers a real-time notification (e.g., a toast message), informing them of the new shared file without requiring a page refresh.

API Design and Documentation

The API is designed following RESTful principles to provide a predictable and standardized interface for client-server communication. Endpoints are resource-oriented, and standard HTTP methods (GET, POST, DELETE) are used for all operations.

Comprehensive API documentation is generated using tools like Swagger/OpenAPI. This provides an interactive way for developers to explore endpoints, understand request/response formats, and test API functionality directly from the browser.

Security Considerations

Security is a foundational aspect of NexusShare, integrated into every layer of the application to protect user data and ensure system integrity.

  • Authentication: Secure user authentication is managed with JSON Web Tokens (JWT). Tokens are signed on the server and have a limited expiration time to prevent misuse.
  • Password Security: All user passwords are put through a one-way hashing algorithm (bcryptjs) before being stored, ensuring that even in the event of a data breach, the raw passwords are not exposed.
  • Access Control: The backend enforces strict authorization checks for every request. Users can only access or modify files and folders they own or for which they have been granted specific permissions.
  • Secure File Handling: File uploads are processed securely to prevent malicious file execution. Public links can be optionally protected with passwords and expiration dates to limit access.

Development Methodology and Tools

The project adopted the Agile Scrum methodology to facilitate iterative development, continuous feedback, and adaptability. Work was organized into two-week sprints, allowing for regular delivery of features and adjustments based on stakeholder input.

  • Version Control: Git was used for source code management, with GitHub serving as the central repository for collaboration and code reviews.
  • Project Management: A tool like Jira or Trello was utilized to manage the product backlog, track sprint progress, and organize tasks.
  • Communication: Slack was the primary tool for team communication, ensuring efficient collaboration and quick resolution of issues.

Testing Strategy

A multi-layered testing strategy was implemented to ensure the quality, reliability, and security of the platform.

  • Unit Testing: Jest and React Testing Library were used to test individual React components and backend utility functions in isolation.
  • Integration Testing: Tests were conducted to verify the interactions between different parts of the application, such as ensuring the frontend correctly communicates with the backend API.
  • End-to-End (E2E) Testing: Automated tools like Cypress were used to simulate complete user workflows, from login and file upload to sharing and logout, ensuring the entire system functions as expected.

Deployment Strategy and Infrastructure

A modern CI/CD (Continuous Integration/Continuous Deployment) pipeline automates the build, test, and deployment process. The platform is designed for cloud-native deployment to ensure scalability and high availability.

  • Containerization: The application is containerized using Docker, which ensures consistency across development, testing, and production environments.
  • Cloud Infrastructure: The platform is deployed on a cloud provider like AWS, utilizing services such as EC2 for compute instances, S3 for file storage, and a managed MongoDB service like Atlas for the database.
  • CI/CD Pipeline: Tools like Jenkins or GitHub Actions are used to automate the deployment pipeline. Every code commit triggers automated tests, and successful builds are deployed to a staging environment before being promoted to production.

Dependencies and Configuration

Key Dependencies

  • Backend: express, mongoose, cors, bcryptjs, jsonwebtoken, multer, socket.io.
  • Frontend: react, react-router-dom, axios, socket.io-client, tailwindcss.

Configuration

Application configuration is managed through environment variables (.env files) for both the backend and frontend. This allows for easy configuration of database URIs, JWT secrets, and API endpoints without hardcoding values.

API Testing Examples

The following `cURL` commands are examples of how to interact with the NexusShare API for common operations.

Register User

curl -X POST http://localhost:5000/api/auth/register \
-H "Content-Type: application/json" \
-d '{"name":"John Doe","email":"john@example.com","password":"password123"}'

Upload File

curl -X POST http://localhost:5000/api/files/upload \
-H "Authorization: Bearer YOUR_JWT_TOKEN" \
-F "file=@/path/to/your/file.pdf"

Share File

curl -X POST http://localhost:5000/api/share/file/YOUR_FILE_ID \
-H "Authorization: Bearer YOUR_JWT_TOKEN" \
-H "Content-Type: application/json" \
-d '{"email":"recipient@example.com","permissions":"download"}'
